Level 0 → 1

Basic introduction to SSH and reading files. Connect to the server, list the directory contents with ls, then read the file with cat.

bash
ssh bandit0@bandit.labs.overthewire.org -p 2220
ls
cat readme
Tip

Use file <filename> to check what type of file something is before trying to read it. Use exit to disconnect.

Level 1 → 2

The file is named -, which bash interprets as stdin rather than a filename. Prefix it with ./ to treat it as a file path.

bash
cat ./-
Note

Running cat - makes bash wait for stdin input. Use Ctrl+D to signal end of file and exit that state.

Level 2 → 3

The file has spaces in its name. Wrap the filename in quotes so the shell treats it as a single argument.

bash
cat "spaces in this filename"

Level 3 → 4

The password is in a hidden file inside the inhere directory. Hidden files start with a dot and aren't shown by default.

bash
cd inhere
ls -la
Lesson learned

When using cd, don't add a leading / for relative paths — cd inhere not cd /inhere. Alternatively, find inhere lists all files including hidden ones without needing to cd first.

Level 4 → 5

Multiple files in inhere, only one is human-readable. Rather than opening each one, use file to check types and find the ASCII text file.

bash
file inhere/*
cat inhere/-file07

Level 5 → 6

The file is somewhere in inhere, human-readable, 1033 bytes, non-executable. Use find with size and type flags.

bash
find inhere -size 1033c -readable ! -executable

Level 6 → 7

File is somewhere on the server, owned by bandit7, group bandit6, 33 bytes. Search from root and filter out permission errors.

bash
find / -size 33c -user bandit7 -group bandit6 2>/dev/null
Tip

Results without Permission denied in front are the accessible ones. The 2>/dev/null redirects error messages so you only see real results.

Level 7 → 8

Password is next to the word "millionth" in data.txt. Use grep to find it.

bash
grep "millionth" data.txt

Level 8 → 9

Password is the only line that appears exactly once in data.txt. Sort first so uniq can detect duplicates, then filter for unique lines.

bash
sort data.txt | uniq -u
Why sort first?

uniq only detects adjacent duplicate lines — sorting groups identical lines together so uniq -u can find the one that's truly unique.

Level 9 → 10

Password is one of the few human-readable strings in a binary file, preceded by several = signs.

bash
strings data.txt | grep "=="
Note

strings extracts printable character sequences from binary files. Combined with grep it narrows down the output efficiently. Alternatively grep -a ==== data.txt works too — the -a flag forces grep to treat binary as text.

Level 10 → 11

The file contains base64-encoded data. Decode it.

bash
base64 -d data.txt
Note

Running base64 without flags encodes. The -d flag decodes.

Level 11 → 12

All letters have been rotated by 13 positions (ROT13). Use tr to reverse it.

bash
cat data.txt | tr 'A-Za-z' 'N-ZA-Mn-za-m'

Level 12 → 13

A hexdump of a file that has been compressed multiple times. Create a working directory, reverse the hexdump, then iteratively decompress.

bash
mktemp -d /tmp/tmp.XXX
cp data.txt /tmp/tmp.XXX/
cd /tmp/tmp.XXX
xxd -r data.txt > data.bin
# Check type, then decompress accordingly:
file data.bin
tar -xf data.bin        # for tar
gzip -d data.bin.gz     # for gzip
bzip2 -d data.bin.bz2   # for bzip2
# Repeat until you get ASCII text
Tip

Use tar -tvf <file> to peek inside a tar archive without extracting. Use file after each extraction to know what to do next.

Level 13 → 14

No password — instead you get an SSH private key to connect directly as bandit14.

bash
ssh -p 2220 -i sshkey.private bandit14@bandit.labs.overthewire.org

Level 14 → 15

Submit the current level's password to port 30000 on localhost. Use netcat.

bash
nc localhost 30000
# paste the password and hit enter

Level 15 → 16

Same as before but the port uses SSL/TLS. Use openssl s_client instead of netcat.

bash
openssl s_client -connect localhost:30001
Why s_client?

s_client initiates a TLS connection and lets you send data interactively — essentially netcat but with TLS support.

Level 16 → 17

Find which port between 31000-32000 speaks SSL and returns credentials. Scan with nmap, then connect with openssl.

bash
nmap -sV -p 31000-32000 localhost
openssl s_client -connect localhost:<correct_port>
Note

If openssl gives KEYUPDATE errors, try ncat --ssl or socat instead — they handle the TLS negotiation differently.

Level 17 → 18

Two password files — find the one line that differs between them.

bash
diff -y passwords.old passwords.new
Note

The -y flag shows both files side by side, making it easy to spot the changed line.

Level 18 → 19

The .bashrc has been modified to log you out immediately on login. Bypass it by not launching bash at all — pass a command directly to SSH.

bash
ssh bandit18@bandit.labs.overthewire.org -p 2220 /bin/dash
Why this works

.bashrc only runs when bash starts. By launching /bin/dash directly instead, we skip bash entirely and the logout trap never fires.

Level 19 → 20

A setuid binary in the home directory lets you run commands as bandit20. Use it to read the password file.

bash
./bandit20-do cat /etc/bandit_pass/bandit20
What is setuid?

A setuid binary runs with the permissions of its owner rather than the person executing it. If the owner is bandit20, running it lets you act as bandit20 for that command.

Level 20 → 21

suconnect connects to a port, expects to receive the current level's password, then returns the next one. You need to be both the server and client — set up a netcat listener first, then run suconnect against it.

bash
# Terminal 1 — start listener with current password
echo "CURRENT_PASSWORD" | nc -lvp 1234

# Terminal 2 — run suconnect
./suconnect 1234

Level 21 → 22

A cronjob is running periodically. Find it in /etc/cron.d and follow the path to see what it does and where it puts the password.

bash
ls /etc/cron.d
cat /etc/cron.d/cronjob_bandit22
cat /usr/bin/cronjob_bandit22.sh

Level 22 → 23

Another cronjob — this one writes the password to a file whose name is derived from a bash script. Read the script, understand what filename it generates, then read that file.

bash
cat /usr/bin/cronjob_bandit23.sh
# The script runs: echo I am user bandit23 | md5sum | cut -d ' ' -f 1
# Run the same command yourself to get the filename
echo I am user bandit23 | md5sum | cut -d ' ' -f 1
cat /tmp/<resulting_hash>
Lesson

Always run commands you find in suspicious scripts to understand what they produce. Never just read them passively.

Level 23 → 24

A cronjob runs and deletes scripts in a directory after executing them. Write a script that copies the password to a readable location, drop it in that directory, and wait.

bash
mkdir /tmp/mydir
cat > /tmp/mydir/myscript.sh <<EOF
#!/bin/bash
cat /etc/bandit_pass/bandit24 > /tmp/mydir/password
EOF
chmod 777 /tmp/mydir/myscript.sh
cp /tmp/mydir/myscript.sh /var/spool/bandit24/foo/
# Wait ~1 minute, then:
cat /tmp/mydir/password
Important

Set permissions correctly with chmod 777 — the cronjob runs as bandit24 and needs to be able to execute your script and write to your directory.

Level 24 → 25

A daemon on port 30002 requires the current password plus a 4-digit PIN. Brute-force all 10000 combinations with a script.

bash
for i in $(seq -w 0000 9999); do
  echo "CURRENT_PASSWORD $i"
done | nc localhost 30002 | grep -v "Wrong"
Why pipe after done?

Piping the entire loop output at once opens only one connection to netcat. If you piped inside the loop you'd open 10000 connections. The -w flag on seq zero-pads numbers so they're always 4 digits.

Level 25 → 26

Bandit26's shell is not /bin/bash — check /etc/passwd to see it's set to more, which reads and exits immediately. To get a shell, shrink your terminal window to just 2 lines so more pauses, then use its v command to open vim, and from vim set the shell and spawn it.

bash
# Shrink terminal to ~2 lines, then SSH in
ssh -i bandit26.sshkey bandit26@localhost -p 2220
# more pauses — press v to open vim
# Inside vim:
:set shell=/bin/bash
:shell

Level 26 → 27

While inside bandit26's shell from the previous level, there's a setuid binary to run commands as bandit27.

bash
./bandit27-do cat /etc/bandit_pass/bandit27

Level 27 → 28

Clone a git repository and read the password from it.

bash
mkdir /tmp/myrepo && cd /tmp/myrepo
git clone ssh://bandit27-git@localhost:2220/home/bandit27-git/repo
cat repo/README

Level 28 → 29

The password in the repo has been redacted. Check the commit history — it was there before.

bash
git log
git show <commit_hash>
Lesson

Sensitive data committed to git is never truly deleted — it lives in the history. git log shows all commits, git show <hash> shows what a specific commit contained.

Level 29 → 30

The password isn't in the main branch. Check other branches.

bash
git branch -a
git checkout <other_branch>
cat README.md

Level 30 → 31

Nothing useful in commits or branches. Check git tags.

bash
git tag
git show <tag_name>

Level 31 → 32

Push a specific file to the remote repository. The .gitignore blocks *.txt — force-add it anyway.

bash
echo "May I come in?" > key.txt
git add -f key.txt
git commit -m "add key"
git push origin master

Level 32 → 33

Dropped into an uppercase shell — every command you type gets uppercased, breaking everything. The trick: use a shell variable that references the shell itself.

bash
$0
# Now in a normal shell
cat /etc/bandit_pass/bandit33
Why $0?

$0 is a special variable that holds the name of the current shell or script. Since it's a variable reference and not a command, uppercasing doesn't affect it — it still expands to the shell binary path and spawns a new normal shell.
